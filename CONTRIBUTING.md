# Contributing

StyleOS Protocol welcomes practical, structured, and privacy-safe contributions.

Before contributing, read:

- [DATA_POLICY.md](DATA_POLICY.md)
- [CONTRIBUTOR_TERMS.md](CONTRIBUTOR_TERMS.md)
- [TRADEMARKS.md](TRADEMARKS.md)

## Contribution Types

### Rule Card

Use a Rule Card to contribute reusable styling logic.

Good Rule Cards include:

- clear applicable conditions;
- concrete recommendation;
- clear avoid list;
- scenario;
- confidence level;
- evidence type;
- source type;
- contributor;
- status.

Use the `Rule Card` GitHub issue template or submit a pull request that follows [Rule Card Schema](docs/rule-card-schema.md).

### Schema Suggestion

Use a Schema Suggestion to improve input fields, tags, report structure, or evaluation criteria.

Good schema suggestions include:

- the current limitation;
- proposed field or tag;
- expected use case;
- compatibility impact;
- example value using synthetic data.

### Synthetic Case

Use Synthetic Case contributions to show how the protocol works without exposing real users.

Synthetic cases must not include:

- real user photos;
- real names;
- real contacts;
- exact addresses;
- private conversation logs;
- unauthorized service records.

### Creator Workflow

Creator Workflow contributions should describe a practical delivery process, such as intake, tagging, rule selection, report writing, review, delivery, feedback, and rule improvement.

### Evaluation Proposal

Evaluation Proposal contributions should improve how the community reviews rule quality, report usefulness, creator fit, consistency, and actionability.

## Pull Request Standard

Before opening a pull request:

- keep changes focused;
- update related links;
- avoid adding external dependencies;
- do not add app code, frontend frameworks, package manifests, build tools, or runtime services;
- confirm examples are synthetic or fully anonymized;
- confirm the PR does not include personal data;
- use clear English headings and Chinese explanations where helpful;
- keep the tone professional, clear, and restrained.

## Review Criteria

Maintainers may review contributions for:

- protocol clarity;
- creator usefulness;
- schema consistency;
- privacy safety;
- commercial boundary clarity;
- trademark safety;
- documentation quality.
