# Why Open Protocol

StyleOS starts as an open protocol because standards should come before apps.

## Standards Before Apps

Before building more interfaces, the field needs shared structures:

- what input should be collected
- how tags should be named
- how rules should express confidence and evidence
- how reports should be reviewed
- how execution cards should be delivered

## Community Before Closed Model

Personal styling is not only a model problem. It includes creator practice, user context, offline execution, feedback, and service design.

An open protocol lets creators and developers improve the structure before everything is hidden inside a closed model.

## Protocol Before Product

The repository defines the workflow language. Future products can build on it.

Open protocol:

- schemas
- starter rules
- synthetic examples
- documentation

Future hosted product:

- dashboards
- accounts
- report generation
- feedback collection
- creator operations

## Open-Core Boundary

Open-core lets StyleOS share enough to build trust and adoption while keeping advanced commercial capabilities sustainable.
